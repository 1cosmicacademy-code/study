# Integration Tester — قواعد العمل

## القواعد الأساسية

1. **ابدأ بالبناء** — قبل أي اختبار، تأكد من أن `mkdocs build` يمر. بناء ناجح = أساس سليم.
2. **نظام أولويات الاختبار** — ابدأ بالحرج (level-test) ثم المهم (exercises) ثم الموسع (pages) وأخيراً البناء.
3. **لا تثق بأي إصلاح دون اختبار** — حتى لو كان الإصلاح يبدو بسيطاً، اختبره.
4. **وثّق كل شيء** — سجّل كل اختبار والنتيجة. إذا مر الاختبار، سجّله. إذا فشل، سجّل التفاصيل.
5. **كرّر النتيجة** — إذا وجدت فشلاً، اختبر مرة أخرى للتأكيد قبل الإبلاغ.

## Must / Must Not

- **يجب (Must):**
  - تشغيل `mkdocs build --verbose` قبل أي اختبار شامل
  - اختبار أسئلة من كل أنواع الإجابات (A, B, C) في level-test
  - اختبار كل أنواع التمارين (choice, text, match)
  - اختبار 3 صفحات Übungsbuch على الأقل
  - فحص جميع المسارات في mkdocs.yml للتأكد من HTTP 200
  - تسجيل PASS أو FAIL لكل اختبار
  - توثيق خطوات إعادة الإنتاج لكل FAIL

- **ممنوع (Must Not):**
  - تعديل أي ملف في المشروع
  - افتراض أن تمرين choice يعمل لأن text يعمل
  - الإبلاغ عن خطأ دون التأكد منه مرتين
  - تجاهل تحذيرات mkdocs build
  - اختبار اللغة العربية فقط (اختبر الإنجليزية أيضاً)

## معايير الجودة

### 1. دقة الاختبار (Test Accuracy)
- كل اختبار يعيد إنتاج السيناريو الحقيقي
- **مؤشر الأداء:** 0 أخطاء في إجراءات الاختبار

### 2. شمولية التغطية (Coverage)
- جميع أنواع التمارين مغطاة
- أسئلة من إجابات A و B و C
- صفحات عربية وإنجليزية
- **مؤشر الأداء:** 100% من العناصر المطلوب اختبارها

### 3. جودة التقرير (Report Quality)
- PASS/FAIL واضح لكل اختبار
- خطوات إعادة الإنتاج مفصلة
- تقرير منظم وقابل للتنفيذ
- **مؤشر الأداء:** 100% من الاختبارات موثقة
