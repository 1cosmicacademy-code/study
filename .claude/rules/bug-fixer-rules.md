# Bug Fix Expert — قواعد العمل

## القواعد الأساسية

1. **الجذر أولاً، العرض ثانياً** — لا تبدأ بالإصلاح قبل تحديد السبب الجذري. قد يكون للعرض الواحد أسباب جذرية متعددة — تأكد أنك تعالج الجذر الصحيح.
2. **أقل تعديل ممكن — maximal impact** — أصلح أقل عدد من الأسطر مع تحقيق أقصى تأثير. سطر واحد صحيح يصلح bug، ثلاثة أسطر خطأ تخلق bugs جديدة.
3. **اختبر الفرضية لا تفترضها** — لا تفترض أنك تعرف سبب الخطأ. ابنِ فرضية → اختبرها → أكّدها أو انفها → ثم ابدأ الإصلاح.
4. **Console هو المصدر الوحيد الموثوق** — أخطاء Console لا تُكذب. إذا Console نظيف والمشكلة قائمة — المشكلة في المنطق وليس في التنفيذ.
5. **لا انحدار (Zero Regression)** — كل إصلاح يجب أن يمر بفحص انحدار: هل الوظائف المجاورة والميزات ذات الصلة لا تزال تعمل؟
6. **وثّق كما لو أن غيرك سيقرأه** — تعليق يشرح "ماذا ولماذا" — ليس "ماذا يفعل الكود" (المطور يقرأ الكود ليفهم ماذا يفعل، التعليق يشرح لماذا تم هذا التعديل).

## Must / Must Not

### يجب (Must):
- قراءة الملف كاملاً قبل التعديل — افهم السياق كاملاً
- تشغيل `mkdocs build --verbose` قبل أي تغيير وبعده — تحقق من عدم كسر البناء
- فحص Console المتصفح لأخطاء JavaScript قبل وبعد الإصلاح
- إضافة تعليق يشرح التغيير (بالإنجليزية أو العربية) يوضح: ما المشكلة، لماذا حدثت، كيف تم إصلاحها
- تغطية الحالات الحدية (edge cases): n=1, empty array, null/undefined, shift=0, special characters
- اختبار الإصلاح منطقياً (trace through the logic line by line)
- اختبار الإصلاح عملياً (شغّل الموقع وجرّب السيناريو)
- تأكيد الخطأ مرتين قبل الإبلاغ عنه — لا تبلغ عن خطأ وهمي
- تصنيف كل خطأ حسب الأولوية: 🔴 حرج, 🟠 عالي, 🟡 متوسط, 🔵 منخفض
- الرجوع إلى `js-developer-rules.md` عند إصلاح أخطاء JavaScript محددة
- الرجوع إلى `exercise-fixer-rules.md` عند إصلاح data-answers في التمارين
- الرجوع إلى `content-fixer-rules.md` عند إصلاح YAML و MkDocs

### ممنوع (Must Not):
- تطبيق إصلاح دون تشخيص مسبق (diagnose قبل fix)
- إعادة كتابة دالة كاملة إذا كان التعديل سطراً واحداً يكفي
- ترك dead code أو commented-out code في الملف بعد الإصلاح
- تجاهل أي Console.error بعد الإصلاح — حتى لو بدا غير مرتبط
- تعديل ملفات لا علاقة لها بالمشكلة الأصلية
- افتراض أن الخطأ يظهر فقط في متصفح واحد — اختبر في log في `mkdocs build`
- تجاهل WARNING في `mkdocs build` بدعوى أنها "ليست ERROR"
- ترك الملفات في حالة "غير قابلة للبناء" — `mkdocs build` يجب أن يمر قبل التسليم
- نسيان فحص النسخة الإنجليزية عند إصلاح مشكلة في النسخة العربية (أو العكس)
- تطبيق إصلاح أمني دون التأكد من أن الوظائف الأساسية لا تزال تعمل

## معايير الجودة

### 1. دقة التشخيص (Diagnosis Accuracy)
- السبب الجذري محدد بدقة: ملف، دالة، سطر، قيمة متغيرة
- **مؤشر الأداء:** 100% من التشخيصات تصيب الجذر من المحاولة الأولى

### 2. دقة الإصلاح (Fix Accuracy)
- الإصلاح يعالج السبب الجذري وليس العرض فقط
- لا يغير سلوك الأجزاء السليمة من الكود
- **مؤشر الأداء:** 100% من الأخطاء تُعالج من جذورها

### 3. الانحدار (Regression)
- 0% أخطاء جديدة ناتجة عن الإصلاح
- Console خالٍ من الأخطاء الجديدة
- `mkdocs build` يمر بنجاح
- **مؤشر الأداء:** 0 regression

### 4. سرعة الإصلاح (Fix Speed)
- تشخيص بسيط: < 5 دقائق
- إصلاح JS متوسط: < 15 دقيقة
- إصلاح بناء: < 10 دقائق
- إصلاح محتوى: < 10 دقائق
- **مؤشر الأداء:** الالتزام بالأوقات حسب تعقيد المشكلة

### 5. جودة التوثيق (Documentation Quality)
- كل إصلاح له تعليق يشرح: الخطأ ← السبب ← الحل
- كل تقرير له: وصف الخطأ، خطوات إعادة الإنتاج، الإصلاح المطبق، التحقق
- **مؤشر الأداء:** 100% من الإصلاحات موثقة

### 6. شمولية التغطية (Coverage)
- جميع أنواع الأخطاء مغطاة: JS, Build, Content, Security
- **مؤشر الأداء:** لا يوجد نوع خطأ غير مغطى في الـ Agent

## المعرفة المطلوبة

### فهم `shuffleOptions()` في level-test.js
```
المشكلة: الدالة تفترض أن qd.answer دائماً 'A' (index 0)
         ثم تحسب الإجابة الجديدة: newIdx = (n - shift) % n
         هذا صحيح فقط لو الإجابة الأصلية index 0

الحل:    اقرأ qd.answer الأصلي ('A', 'B', 'C')
         حوّله إلى index: origIdx = OPTION_LABELS.indexOf(qd.answer)
         طبّق shift: newIdx = (origIdx + shift) % n
         ثم qd.answer = OPTION_LABELS[newIdx]
```

### فهم `exercise-checker.js` — مقارنة الإجابات
```
setupChoice(exerciseDiv, data):
  - يولد radio buttons لكل خيار
  - radio.value = String.fromCharCode(65 + j - 2) → j=2 → "A", j=3 → "B"

checkChoice(exerciseDiv, data):
  - selectedValue = radio.value  // "A"
  - correct = selectedValue === data.answers[i]
  
→ data-answers يجب أن تكون A/B/C وليس أ/ب/ج
```

### فهم `video-switcher.js`
```
- يبحث عن h2 يحتوي "فيديو"/"Video"
- ينشئ iframe يوتيوب
- مشاكل محتملة: h2 غير موجود، videos.json لا يُحمّل، استثناء يوقف التنفيذ
```

### أخطاء `mkdocs build` الشائعة
```
ERROR - Error reading page: YAML front matter → تحقق من مسافات front matter
ERROR - Config value: 'theme' → تحقق من theme name في mkdocs.yml
WARNING - A relative path ... is not a valid URL → تحقق من صحة المسار
WARNING - Document ... is not included in the navigation → صفحة يتيمة
```

## أنماط الأخطاء المعروفة في المشروع

### Pattern 1: JS الإجابة الأصلية محسوبة بافتراض index 0
- **الموقع:** `docs/assets/js/level-test.js` — `shuffleOptions()`
- **الوصف:** الدالة تفترض أن الإجابة الأصلية دائماً 'A' قبل الخلط
- **الإصلاح:** قراءة `qd.answer` الفعلي (`'A'`, `'B'`, `'C'`) بدلاً من افتراض index 0

### Pattern 2: data-answers بحروف عربية في تمارين choice
- **الموقع:** `docs/a1/ubungsbuch/*.md`, `docs/grammatik/*.md`
- **الوصف:** data-answers تحتوي `أ/ب/ج` بدل `A/B/C`
- **الإصلاح:** تحويل الحروف العربية إلى لاتينية (أ←A, ب←B, ج←C)

### Pattern 3: خلط اللغات في التمارين الإنجليزية
- **الموقع:** `docs/**/*.en.md`
- **الوصف:** كلمات إنجليزية تظهر في أعمدة "Words" التي يجب أن تكون بالألمانية
- **الإصلاح:** استبدال الكلمات الإنجليزية بالألمانية الصحيحة

### Pattern 4: روابط مقطوعة في صفحات القواعد
- **الموقع:** `docs/grammatik/*.md`
- **الوصف:** مسارات نسبية تشير لملفات غير موجودة أو بأحرف خاطئة
- **الإصلاح:** تصحيح المسار (حساسية حالة الأحرف مهمة!)

### Pattern 5: مكتبات خارجية بدون SRI
- **الموقع:** `docs/overrides/*.html` أو `mkdocs.yml`
- **الوصف:** نصوص CDN محملة بدون integrity hash
- **الإصلاح:** إضافة integrity attribute للـ `<script>` و `<link>` الخارجية

### Pattern 6: خطأ في عدد data-answers
- **الموقع:** `docs/**/*.md` — تمارين تفاعلية
- **الوصف:** عدد عناصر data-answers لا يساوي عدد صفوف tbody tr
- **الإصلاح:** إضافة/إزالة القيم للمطابقة

## قائمة التحقق قبل التسليم

### للـ JS Fixes
- [ ] هل قرأت الملف كاملاً وفهمت تدفق البيانات؟
- [ ] هل حددت السبب الجذري بدقة (دالة، سطر، قيمة)؟
- [ ] هل طبّقت أقل تعديل ممكن؟
- [ ] هل أضفت تعليقاً يشرح التغيير؟
- [ ] هل اختبرت الإصلاح منطقياً (trace)؟
- [ ] هل اختبرت الإصلاح عملياً (Console, build)?
- [ ] هل Console خالٍ من الأخطاء الجديدة؟
- [ ] هل جميع الحالات الحدية مغطاة؟
- [ ] هل `mkdocs build --verbose` يمر؟

### للـ Build Fixes
- [ ] هل `mkdocs build --verbose` يمر بدون ERROR؟
- [ ] هل جميع WARNING مبررة وليست خطيرة؟
- [ ] هل الصفحات العربية لا تزال RTL؟
- [ ] هل الصفحات الإنجليزية لا تزال LTR؟
- [ ] هل جميع الصفحات تُولد في `site/`؟

### للـ Content Fixes
- [ ] هل `mkdocs build --verbose` يمر؟
- [ ] هل النسختان العربية والإنجليزية متناسقتان؟
- [ ] هل خلط اللغات مُصحَّح؟
- [ ] هل الروابط الداخلية تعمل؟
- [ ] هل الجداول منسقة بشكل صحيح؟

### للـ Security Fixes
- [ ] هل CSP policy تسمح بالموارد الضرورية فقط؟
- [ ] هل جميع النصوص الخارجية لها SRI؟
- [ ] هل Console خالٍ من mixed content warnings؟
- [ ] هل التمارين لا تزال تعمل بعد التعديلات الأمنية؟
- [ ] هل الموقع لا يزال صالحاً للتشغيل المحلي (mkdocs serve)؟

### للـ Verify (فحص الانحدار)
- [ ] هل الخطأ الأصلي لم يعد يظهر؟
- [ ] هل الميزات المجاورة تعمل؟
- [ ] هل `mkdocs build` يمر؟
- [ ] هل Console خالٍ من الأخطاء الجديدة؟
- [ ] هل النسختان العربية والإنجليزية تعملان؟
