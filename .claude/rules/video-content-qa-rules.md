# قواعد مراجعة جودة الفيديو (Video Content QA)

## الأساسيات (Must)
1. **قراءة الدرس أو القاعدة بالكامل** – يجب فحص ملف `.md` أو `.en.md` لتحديد هدف الدرس ومستوى CEFR (A1/A2).
2. **استخراج معرف الفيديو** – البحث عن السمة `data-video-id` داخل ملف الـ Markdown.
3. **تحقق من صلاحية الرابط** – طلب HEAD إلى `https://www.youtube-nocookie.com/embed/<VIDEO_ID>` والتأكد من استجابة HTTP 200.
4. **تطابق المحتوى** – تشغيل Kimi WebBridge ومراجعة أول 30 ثانية من الفيديو للتأكد من مطابقة العنوان، الوصف والكلمات المفتاحية مع محتوى الدرس أو القاعدة.
5. **جودة الإنتاج** – التأكد من:
   - دقة الصورة ≥ 720p
   - وضوح الصوت وخلوه من الضوضاء
   - مدة الفيديو من 2 دقيقة إلى 10 دقائق (حسب الموضوع).
6. **مقاييس التفاعل** – يجب أن تكون نسبة الإعجابات ≥ 95 % من مجموع التقييمات، وعدد المشاهدات ≥ 10 k (أو ≥ متوسط الفئة إذا كان الحد أخر).
7. **تحديث `videos.json`** – إذا كان الفيديو غير صالح أو يحتاج لتحديث، تعديل الحقل `status` إلى `invalid` أو `needs-refresh` مع تاريخ الفحص.
8. **تشغيل `mkdocs build --verbose`** بعد أي تعديل على ملف الـ Markdown أو `videos.json` لضمان عدم وجود أخطاء بناء.
9. **توثيق السبب** – إضافة تعليق داخل الملف:
   ```html
   <!-- video‑qa‑rationale: تم استبدال الفيديو بسبب عدم تطابق المحتوى أو ضعف الجودة -->
   ```
10. **إشعار فوري** – إذا تم اكتشاف فيديو حرج (broken embed or off‑topic) يجب إرسال **PushNotification** للمسؤول.

## الممنوعات (Must Not)
- تعديل أي ملفات JavaScript أو CSS (`video-switcher.js`, `exercise-checker.js` …).
- حذف فيديو موجود إذا لم يتم توثيق السبب أو تقديم بديل مناسب.
- تجاهل تحذيرات `mkdocs build` أو الاستمرار في بناء فاشل.
- الاعتماد على رأي شخصي فقط دون تحقق بوسائل الـ WebBridge.
- ترك أي فيديو غير صالح في `videos.json` دون تغيير الحقل `status`.

## مؤشرات الأداء (KPIs)
| KPI | الهدف |
|-----|-------|
| **نسبة صلاحية الروابط** | 100 % روابط HTTP 200 |
| **مستوى تطابق المحتوى** | ≥ 90 % وفقًا لتقييم Kimi WebBridge |
| **جودة الإنتاج** | ≥ 720p، صوت واضح، مدة 2‑10 دقيقة |
| **نسبة الإعجابات** | ≥ 95 % من مجموع التقييمات |
| **عدد المشاهدات** | ≥ 10 k أو ≥ متوسط الفئة |
| **وقت الفحص** | ≤ 5 دقائق لكل فيديو (HTTP HEAD + WebBridge preview) |
| **معدل الاستبدالات** | ≤ 2 % من إجمالي الفيديوهات شهريًا |
| **إشعارات الأخطاء الحرجة** | يتم إرسال PushNotification خلال 1 دقيقة من الاكتشاف |

## أوضاع التشغيل (Operating Modes)
| الوضع | الوصف |
|------|------|
| **audit** | فحص جميع الفيديوهات في مسار محدد أو في جميع صفحات الدروس والقواعد. يولد تقرير JSON شامل.
| **verify** | فحص فيديو واحد تم تمريره عبر معرف `<VIDEO_ID>` أو مسار ملف. يعيد `true/false` مع تفاصيل الفشل إن وجد.
| **report** | إنشاء تقرير markdown (`/docs/qa/video-report-YYYY-MM-DD.md`) يتضمن جدولًا بكل الفيديوهات مع حالة الفحص، السبب، وتوصية الإجراء.
| **alert** | إرسال PushNotification فوري عند اكتشاف فيديو غير صالح أو غير مطابق.

## تكامل النظام
- يعتمد على **video-switcher.js** لتضمين الفيديو عبر السمة `data-video-id` داخل الـ Markdown.
- يستخدم ملف **videos.json** لتخزين ميتادات الفيديو (العنوان، القناة، المشاهدات، الإعجابات، الحالة).
- لا يجوز تعديل أكواد الـ JS – يقتصر الدور على مراجعة المحتوى والبيانات وتحديث `videos.json`.

## ملاحظات تنفيذية
- جميع عمليات الفحص تُجرى داخل بيئة **Sonnet**.
- يجب تشغيل **mkdocs build --verbose** بعد كل تعديل لضمان عدم كسر البناء.
- جميع النتائج يجب إرجاعها بصيغة JSON موحدة (انظر مثال في ملف الـ Skill).
