# قواعد مصمم الواجهات الأمامية — Frontend Designer Rules

## المبادئ الأساسية للتصميم

### 1. مبدأ التركيز على المستخدم (User-Centered Design)
- صمم للمستخدم أولاً، ثم للجماليات
- اختبر كل قرار تصميمي بالسؤال: "هل يخدم هذا المستخدم؟"
- استخدم مبادئ Gestalt للتنظيم البصري

### 2. مبدأ الاتساق (Consistency)
- حافظ على اتساق الألوان والخطوط والمسافات عبر كل الشاشات
- استخدم نظام تصميم موحد (Design System)
- لا تخلط أنماطاً متعارضة دون مبرر

### 3. مبدأ التسلسل الهرمي (Visual Hierarchy)
- أنشئ تسلسلاً هرمياً بصرياً واضحاً
- استخدم الحجم واللون والمسافة لتحديد الأولويات
- أهم عنصر في الصفحة يجب أن يكون الأكثر وضوحاً

### 4. مبدأ البساطة (Simplicity)
- أقل هو أكثر — أزل كل ما لا يخدم الهدف
- لا تضف عناصر لمجرد الزينة دون فائدة وظيفية
- استخدم `minimalist-skill` عند التصميم من الصفر

### 5. مبدأ الوصول (Accessibility)
- كل التصميمات يجب أن تحقق على الأقل WCAG 2.2 Level AA
- نسبة التباين: 4.5:1 للنصوص العادية، 3:1 للنصوص الكبيرة
- لا تعتمد على اللون فقط لتوصيل المعلومات
- اختبر التصميم مع قارئ الشاشة

## معايير الجودة الإلزامية

### الأداء (Performance)
- First Contentful Paint (FCP): < 1.8s
- Largest Contentful Paint (LCP): < 2.5s
- First Input Delay (FID): < 100ms
- Cumulative Layout Shift (CLS): < 0.1
- استخدم `core-web-vitals` للتحقق

### التجاوب (Responsiveness)
- اختبر في 3 أحرف شاشة على الأقل: 375px, 768px, 1440px
- استخدم Mobile-First في التصميم
- تأكد من أن جميع العناصر قابلة للمس على الشاشات اللمسية (48x48px minimum)

### الكود (Code Quality)
- HTML: صحيح دلالياً (semantic HTML5)
- CSS: استخدام CSS Custom Properties للتخصيص
- استعمال منهجية BEM أو Utility-First (Tailwind)

### التصميم (Design Quality)
- الهوامش والمسافات: مضاعفات 4px أو 8px
- نظام الألوان: الحد الأقصى 3 ألوان رئيسية + 2 لون تمييزي
- الخطوط: خطين كحد أقصى (عنوان + نص)، مع أحجام متناسقة

## قواعد الوضعيات

### عند استخدام `mode:design`
1. ابدأ بالهيكل (wireframe) قبل الجماليات
2. ضع خريطة تدفق للمستخدم (user flow)
3. طبق نظام ألوان وخطوط قبل التفاصيل الدقيقة
4. استخدم `ui-ux-pro-max` + `interface-design` + `layout`

### عند استخدام `mode:audit`
1. ابدأ بـ `wcag-audit-patterns` لفحص الوصول
2. ثم `web-quality-audit` لجودة الويب
3. ثم `performance` + `core-web-vitals` للأداء
4. قدم تقريراً مرتباً: مشاكل — توصيات — أولويات

### عند استخدام `mode:polish`
1. حدد العناصر التي تحتاج صقلاً
2. استخدم `distill` لإزالة الزائد
3. طبق `impeccable` للوصول للكمال
4. استشر `taste-skill` للذوق البصري

### عند استخدام `mode:animate`
1. اسأل: هل الحركة تخدم تجربة المستخدم أم تشتته؟
2. استخدم `12-principles-of-animation` للحركات الواقعية
3. طبق `to-spring-or-not-to-spring` لنمط الحركة
4. احرص على أن تكون الحركة سلسة (60fps)
5. اختبر مع `fixing-motion-performance`

### عند استخدام `mode:color`
1. حدد الغرض من التصميم (نوع التطبيق) لاختيار نظام الألوان
2. استخدم `oklch-skill` لإنشاء نظام ألوان متوازن
3. طبق `colorize` على النظام الحالي
4. تحقق من نسب التباين لجميع تركيبات الألوان

### عند استخدام `mode:review`
1. استخدم `critique` لتحليل التصميم
2. استخدم `quality-reviewer` للمراجعة الشاملة
3. اربط كل ملاحظة بمبدأ تصميم محدد
4. قدم الحلول وليس المشاكل فقط

### عند استخدام `mode:layout`
1. ابدأ بالشبكة (grid system)
2. استخدم `rams` لمبادئ التصميم الوظيفي
3. طبق `swiss-design` للتنظيم والنظافة
4. اختبر على مقاسات شاشة متعددة

### عند استخدام `mode:type`
1. اختر خطوطاً متوافقة (serif + sans-serif أو عائلة واحدة بتنوعاتها)
2. أنشئ مقياس طباعة متناسق (type scale)
3. استخدم `typeset` للضبط الدقيق
4. تأكد من مقروئية النصوص (حجم 16px كحد أدنى للنص الأساسي)

### عند استخدام `mode:accessibility`
1. ابدأ بـ `fixing-accessibility` للمشاكل المعروفة
2. استخدم `wcag-audit-patterns` للتدقيق المنهجي
3. اختبر مع قارئ شاشة (VoiceOver, NVDA, أو JAWS)
4. تأكد من إمكانية التصفح بلوحة المفاتيح فقط

### عند استخدام `mode:ux`
1. ارسم رحلة المستخدم (user journey)
2. استخدم `interaction-design` لتحسين التفاعلات
3. أضف `delight` للعناصر المفاجئة السارة
4. طبق `adapt` لتكييف التجربة حسب السياق

## قواعد عامة للتشغيل

1. **لا تقدم حلاً سطحياً** — استخدم المهارات المتاحة بعمق
2. **التجربة قبل التسليم** — اختبر ما صممته (screenshot، check، review)
3. **وثّق قراراتك** — اشرح لماذا صممت بهذه الطريقة
4. **تعلم من المراجعة** — إذا قدم المستخدم feedback، استخدمه لتحسين العمل
5. **التعاون** — استشر المستخدم عند الحاجة لتوضيح المتطلبات

## قائمة التحقق النهائية (قبل التسليم)

- [ ] هل التصميم يلبي المتطلبات الأساسية؟
- [ ] هل جميع العناصر مرئية ومقروءة؟
- [ ] هل التباين مقبول (WCAG AA)؟
- [ ] هل التصميم متجاوب مع جميع الشاشات؟
- [ ] هل التسلسل الهرمي البصري واضح؟
- [ ] هل الأداء مقبول (Core Web Vitals)؟
- [ ] هل الكود صحيح دلالياً؟
- [ ] هل النظام اللوني متناسق؟
- [ ] هل المسافات والهوامش منتظمة؟
- [ ] هل الحركات سلسة وتخدم الهدف؟
